## V1-V2迁移概述

从官方迁移结构看，V1 到 V2 不是简单的装饰器改名，而是状态建模方式、观察机制和组件通信方式一起变化。

迁移时建议按下面顺序理解：

1. 先区分“组件内状态变量”“数据对象状态变量”“应用内状态变量”三类问题。
2. 再判断当前是直接迁移到纯 V2，还是需要短期保留 V1/V2 混用。
3. 最后才选择具体 API，如 `@Local`、`@Param`、`@Event`、`@ObservedV2`、`@Trace`、`makeObserved`。

## 状态管理V1向V2迁移场景

官方把 V1 向 V2 的迁移重点拆成了组件内状态、数据对象、应用内状态、组件复用、循环渲染、内置对象和动画几个子场景。本仓库当前优先离线覆盖最常见、且最容易断链的几个部分。

### 组件内状态变量迁移

常见映射关系可以先按下面理解：

- `@State`:
  不需要外部初始化时，优先迁移为 `@Local`；需要外部初始化一次时，迁移为 `@Param` 或 `@Once`。
- `@Prop`:
  V2 中通常改为 `@Param`，但要注意 V1 的深拷贝语义与 V2 引用传递语义不同。
- `@Link`:
  V2 中通常显式拆为 `@Param + @Event`，不再依赖框架内建的双向同步语义。
- `@Provide/@Consume`:
  对应迁移为 `@Provider/@Consumer`。
- `@Watch`:
  对应迁移为 `@Monitor`，但监听触发时机和聚合方式不同。
- 重复计算:
  V2 可以用 `@Computed` 收敛重复计算逻辑。
- `$$` 双向绑定:
  在 V2 中改为 `!!` 语法。

### 数据对象状态变量迁移

官方把这部分单独拆成一页，核心结论是：

- V1 常见组合是 `@ObjectLink + @Observed + @Track`。
- V2 的主组合是 `@ObservedV2 + @Trace`。
- `@ObservedV2` 代表“这个 class 可以被观察”，真正需要精确跟踪的属性仍要落在 `@Trace` 上。

如果迁移目标只是让对象内部字段在 V2 中可刷新，优先判断是应该把 class 迁移到 `@ObservedV2/@Trace`，还是只需要对运行时对象使用 `makeObserved`。

## 什么时候需要 makeObserved

以下场景通常需要优先考虑 `UIUtils.makeObserved(...)`：

- 数据源来自普通 class、`JSON.parse()` 返回值或运行时拼装对象，本身不具备 V2 可观察能力。
- 组件依赖对象内部字段刷新，而不是只依赖对象引用整体替换。
- 列表、瀑布流、动态属性等场景需要感知对象内部字段变化。

不建议为了迁移而无差别包裹所有对象。只有确实需要让 V2 观察内部属性变化时，再对目标数据调用 `makeObserved`。

### 组件内状态变量迁移与 makeObserved 的边界

`makeObserved` 不是装饰器迁移的直接替代物。它更适合“对象已经是普通对象，但又需要在 V2 中观察内部字段变化”的场景；而 `@Local`、`@Param`、`@Event`、`@ObservedV2`、`@Trace` 属于状态建模层面的正式迁移方案。

## 滚动组件

`List`、`WaterFlow` 这类滚动组件常见问题，是数据项内部字段已经变化，但界面没有触发刷新。出现这种情况时，需要优先检查列表项对象是否仍是普通对象。

建议按下面顺序处理：

1. 明确列表项的数据源类型，确认是否为普通 class、普通 object 或反序列化数据。
2. 在进入组件树之前，将需要被观察的数据转换为 `makeObserved` 返回值。
3. 只让真正参与界面渲染的对象进入可观察链路，避免把无关大对象整体代理。

可直接对照以下组件文档中的示例：

- [List](../topics/components/List.md)
- [WaterFlow](../topics/components/WaterFlow.md)

## modifier

动态属性或 `Modifier` 场景中，如果按压态、选中态等依赖对象内部字段变化，而字段变化后组件没有重新应用属性，通常也是因为传入的对象不是 V2 可观察对象。

建议做法：

1. 保持 `Modifier` 本身职责单一，只接收需要渲染的状态。
2. 对传入 `Modifier` 的普通对象使用 `makeObserved`，确保内部字段变化可被跟踪。
3. 如果只是简单布尔态或枚举态，优先继续使用基础状态变量，避免引入不必要的对象层级。

相关组件文档：

- [动态属性设置](../topics/components/动态属性设置.md)

## 状态管理V1和V2混用场景

如果页面仍保留 V1 状态对象，但局部子树已经改成 `@ComponentV2`，不要继续硬套纯 V2 迁移思路，应改看 [状态管理V1和V2混用指导（API version 19及之后）](状态管理V1和V2混用指导（API%20version%2019及之后）.md)。

## 迁移建议

- 新代码优先使用状态管理 V2 模式，减少 V1 与 V2 长期混用。
- 迁移时先定位“是否需要观察对象内部字段”，再决定是否使用 `makeObserved`。
- 当页面仍保留 V1 状态对象但子树已迁入 `@ComponentV2` 时，改用 [状态管理V1和V2混用指导（API version 19及之后）](状态管理V1和V2混用指导（API%20version%2019及之后）.md) 中的兼容方案。
