# 应用模型与 ArkTS 并发模型

本文为基于 HarmonyOS 官方文档整理的主题导读，便于快速理解应用模型与并发机制。

HarmonyOS 提供了两种主要的应用模型，并在 ArkTS 语言中设计了独特的并发机制。

## 1. 应用模型 (Application Models)

应用模型定义了应用的生命周期、组件类型以及进程结构。

### Stage 模型（主推）
自 HarmonyOS 3.1 开始主推的模型，专为复杂应用和多设备协同设计。
*   **核心理念**：将应用组件（如 UI 展示、后台服务）与运行进程进行了解耦。
*   **组件分类**：
    *   **UIAbility**：包含 UI 界面的组件，负责与用户交互。一个应用可以有多个 UIAbility。
    *   **ExtensionAbility**：面向特定场景的无界面或部分界面的扩展能力（如输入法、卡片、后台服务）。
*   **优势**：
    *   **进程模型清晰**：系统可以根据内存情况将多个 UIAbility 放在同一个进程，或分别放在不同进程。
    *   **逻辑解耦**：全局的 `AbilityStage` 和局部的 `UIAbility` 生命周期分离，有利于状态共享。
    *   **多设备支持**：天然支持跨端流转。

### FA 模型 (Feature Ability)
早期版本的模型，结构类似于 Android 的 Activity 体系，主要包含 PageAbility、ServiceAbility 等。
*   **状态**：目前已不推荐在新项目中使用，仅作兼容性维护。所有新开发（特别是 API 12+ / HarmonyOS NEXT）**必须使用 Stage 模型**。

## 2. ArkTS 并发模型
ArkTS 摒弃了传统的共享内存多线程（如 Java 的 Thread），采用了**消息传递**的并发模型（Actor 模型），以避免死锁和数据竞争。

### 单线程执行与 Event Loop
*   每个 ArkTS 实例（如主线程）都有自己的 Event Loop。
*   所有的 UI 渲染、事件响应都在**主线程（UI 线程）**执行。
*   **铁律**：绝对不要在主线程执行耗时的 I/O 或 CPU 密集型任务，这会导致丢帧和卡顿。

### 多线程解决方案
如果需要执行耗时任务，ArkTS 提供了两种并发能力：

1.  **TaskPool (任务池 - 推荐)**
    *   **特点**：轻量级。开发者只需将任务（Task）提交给 TaskPool，系统会自动管理工作线程的创建、分配和销毁。
    *   **适用场景**：大量独立的、不需要常驻的耗时计算（如图片解码、JSON 解析）。
    *   **限制**：任务函数必须使用 `@Concurrent` 装饰器，且参数传递存在类型限制。

2.  **Worker**
    *   **特点**：重量级。开发者需要手动创建、管理生命周期，Worker 线程会一直运行直到被主动终止。
    *   **适用场景**：长时间运行的后台服务（如持续的音频解码、WebSocket 长连接维持）。
    *   **通信**：主线程与 Worker 之间通过 `postMessage` 和 `onmessage` 进行序列化的数据通信。
